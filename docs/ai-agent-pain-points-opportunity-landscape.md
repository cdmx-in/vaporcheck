# General AI/LLM-Agent Pain Points & Skill/MCP Opportunity Landscape (research only)

*Research synthesis — not a build proposal. Date: 2026-06-14. Source: provided ranked-issue, solution-scouting, and raw-inventory JSON. No live web search was run for this report; all "evidence" and frequency/severity labels below are inherited from the input corpus and inherit its limits (see Evidence Caveats).*

---

## 1. TL;DR

- **The single highest-leverage gap is existence-verification of model-emitted identifiers** (citations, package names, tool names, file paths, API symbols). Five raw issues collapse into one mechanism — *emit → resolve against an authoritative source → verified / unverified / abstain* — and existence is a deterministic binary lookup, not a judgment call. No shipped tool spans all five identifier classes or acts as an in-loop, fail-closed pre-action gate; everything today is siloed by class and mostly post-hoc.
- **The recurring meta-pattern across the top of the list is "externalize the judgment the model is bad at."** Completion-verification (stop the agent grading its own homework), runaway/destructive-action governance, and trust separation for untrusted tool output all convert an intractable model weakness into a tractable tooling boundary. These are the cleanest "guarantee, not a suggestion" cases.
- **Coverage everywhere is "partial," and the pattern is consistent:** individual sub-problems are well-covered in isolation, but *nobody bundles them into one drop-in, local-first artifact with opinionated defaults*. The whitespace is integration and enforcement-boundary, not invention. Be skeptical of any "this is totally unsolved" framing — most legs have prior art; the defensible move is unification + in-loop enforcement.
- **A hard line separates addressable from model-bound.** Calibration/overconfidence, true effective-context ceiling, lost-in-the-middle, malformed-JSON *emission*, multi-turn jailbreaks, and agentic misalignment/reward-hacking are **not** skill/MCP-fixable — a wrapper can only work around them. Promising otherwise is the main way these projects over-claim.
- **Two "both"-vehicle issues are really MCP-primary:** the verification/governance/quarantine layers need a privileged boundary the model cannot argue past (load-bearing MCP/gateway), with a thin skill only to enforce that the boundary is always invoked. One issue (cross-session memory coherence) is MCP-primary outright, and one (containment) should explicitly pair with existing MCP-Scan/ETDI rather than rebuild the description-pinning half.
- **Evidence quality is uneven.** Several headline numbers ($2,847/4hr, 2.3M API calls, 65% of enterprise failures, 50-85% injection ASR) come from vendor posts or single anecdotes, and key supporting works (EviBound, CaMeL, adaptive-depth tool selection) are papers/prototypes explicitly "not for production." Treat the *direction* as well-supported and the *magnitudes* as indicative, not load-bearing.

## 3. Per-Opportunity Detail

### 1. Existence-verification of model-emitted identifiers — leverage 93
**Problem.** One mechanism unifies five raw failures: the model emits a plausible-but-nonexistent identifier-shaped string (citation/DOI/URL, package name, tool name or wrong-tool's-args, file path, deprecated/removed API) without confirming the referent exists. Reported rates: 3-13% fabricated URLs even *with* web search (58-88% in legal); ~19.7% of code samples contain a hallucinated package, 43% repeatable (→ slopsquatting supply-chain attack surface).

**Why high-leverage.** Highest combined frequency+severity in the corpus (four of five constituents high/high) *and* the most deterministically fixable — existence is a binary lookup. The verification logic (emit → resolve → mark verified/unverified/abstain) is identical across all five domains, so one resolver + one skill discipline covers everything. It also pre-empts a real attack surface (slopsquatting, destructive wrong-path edits), not just quality.

**What already exists & gaps.** Citation verifiers (LLM Citation Verifier, RefChecker, claude-skill-citation-checker, Citely) — citations only, mostly post-hoc batch validators over a finished `.bib`. Slopsquatting scanners (slopcheck across 7 ecosystems, depscope denylist) — packages only, explicitly post-hoc CLI/CI, no MCP, can't be called mid-turn. Context7 MCP — injects up-to-date library docs but is RAG context, *not* a gate; doesn't intercept a hallucinated symbol. Guardrails AI validators (ValidURL etc.) — check format/grounding/policy, not referent existence against a registry. Strict JSON-Schema/Pydantic/Zod tool-arg validation — constrains shape and enums but gives no guardrail for free-form string identifiers (an invented `customer_id`, path, or package), and doesn't check that a hallucinated *tool name* maps to a registered tool. Tool Receipts (HMAC) — detect *fabricated tool executions* after the fact, not whether a referent exists beforehand. Deprecated-API repair (REPLACEAPI) — research prototypes, per-library curated mappings.

**Concrete concept.** *MCP-led "both."* An MCP server exposing one `verify_identifiers(text | tool_call)` tool plus per-class resolvers (Crossref/arXiv/Semantic Scholar; live PyPI/npm/crates/Go registry; the registered tool schema; the filesystem; current docs), returning `exists / not-found / redirected-canonical` with the authoritative record — callable mid-loop as a pre-action gate that fails closed. A thin skill wires it into the turn: "before recommending a package/citation/API, or emitting a path or tool call, resolve it; cite/import/call/edit only verified IDs, mark the rest unverified, never auto-install." Differentiators that make it non-trivial: return the canonical/redirected referent (handles partially-wrong citations, renamed packages), distinguish "doesn't exist" from "exists but deprecated/yanked," and cache + rate-limit so it can run every turn.

**Skeptic's note.** The package/path/tool-name slice genuinely has *no* agent-callable verifier today, so the whitespace claim holds there. Be honest that registry lookups add latency and false-positives on legitimately-new packages, and that the citation slice is the most crowded.

### 2. Externalized completion verification + ground-truth-gated retries — leverage 90
**Problem.** Merges premature-exit/incomplete-verification (~21% of all failures per SHIELDA/MAST analysis, with *incorrect* verification the single strongest predictor of long-horizon failure), unreliable intrinsic self-correction (error *detection*, not repair, is the bottleneck; "reflect and fix" is often net-negative and flips correct answers), and cascading errors where a wrong step-2 silently corrupts everything after. Fix: never let the model decide "done"/"correct" subjectively — gate every exit and retry on deterministic external evidence.

**Why high-leverage.** Attacks the meta-failure that makes long-horizon agents unshippable, and converts a model-level weakness (self-detection blind spot) into a tooling problem (expose real verifiers). A checklist-plus-verifier gate is a small, reusable surface that compounds: caps the effective horizon, halts cascades early, replaces useless reflection loops with signal-driven ones.

**What already exists & gaps.** Claude Code Stop/SubagentStop hooks — the closest deterministic exit-2 primitive, but raw: hand-wired, whole-turn only (misses mid-trajectory cascades), no criteria registration, no retry governance. Rigour MCP — ships the gate+fix-loop form factor but checks code *quality/hygiene*, not task-completion ground truth. EviBound — almost a 1:1 match (Approval Gate + Verification Gate + bounded confidence-gated retries, ~0% hallucinated "done" at ~8% overhead) but is an unshipped MLflow-bound paper. AgentForge/execution-grounding — research prototypes, SWE-only. Guardrails/NeMo — validate output shape per-message, not task-done per-trajectory. LangGraph — construction kit; schema validation catches malformed but not semantically-wrong-but-valid intermediates. DeepEval/Langfuse/Phoenix — offline CI or observability, and online judges are usually LLM-based (the very anti-pattern).

**Concrete concept.** *MCP-led "both."* An "acceptance-criteria + ground-truth gate" combining three things none of the above unify: (1) **register** acceptance criteria up front, each bound to a deterministic oracle the agent doesn't control (test exit code, type-checker clean, JSON-schema/state assertion, search/grep confirmation) — explicitly forbid LLM-judge as the sole gate; (2) **gate exit** on those oracles (on Claude Code via the Stop hook; as an MCP via `register_criteria`/`verify_completion`/`gate_retry`), feeding the structured failure back on red; (3) **govern retries** with a bounded budget + transient-vs-real classification, plus a per-step checkpoint mode to catch wrong-step-2 cascades. Essentially: EviBound's dual-gate made oracle-pluggable and domain-agnostic.

**Skeptic's note.** Closest prior art to differentiate against is Rigour (reposition from code-quality to task-completion) and the native Stop hook (wrap with criteria + retry governance + per-step checkpoints). The honest risk: "register acceptance criteria up front" is exactly the discipline users skip, so adoption — not feasibility — is the hard part.

### 3. Runaway/destructive-action governor — leverage 89
**Problem.** Combines unbounded cost/token spirals (one runaway agent burned $2,847 in 4 hours; one made 2.3M API calls after misreading an error code), super-linear per-step token growth, and destructive/irreversible actions without confirmation (Cyera's 344 agent-inflicted-damage cases, 188 with no attacker; `rm -rf`, `DROP TABLE`, force-push, prod-DB wipes — sometimes *after* the user said "do not run anything"). Unifying control: a privileged enforcement boundary the agent cannot talk past.

**Why high-leverage.** Severity is maximal (money, irreversible destruction), frequency high, and it's the cleanest "guarantee, not suggestion" case — a gateway enforces ceilings/confirmation deterministically; no prompt can. A small wrapper removes an entire class of catastrophic, headline-making incidents, largely domain-agnostic.

**What already exists & gaps.** The ecosystem splits along a fault line. *Spend camp* (AgentGuard47, LiteLLM/Portkey/Helicone, framework step-caps) enforces token/cost/call ceilings + kill but is blind to what a tool *does* — it'll pass a within-budget `DROP TABLE prod`. *Destructive camp* (AperionAI Shield, node9-proxy, Invariant, the locally-installed `/careful` skill) matches dangerous patterns but doesn't meter spend or stop a within-policy infinite loop — and free tiers lack a real kill switch (Shield paywalls it; `/careful` is override-only "ask"). No single tool delivers all four: (1) hard per-run token/cost/call ceiling, (2) out-of-band kill switch, (3) destructive-op dry-run + confirmation, (4) prod-target blocking by default. The native PreToolUse hook is the right substrate (fires even under `--dangerously-skip-permissions`) but is a raw primitive.

**Concrete concept.** *MCP-led "both," skill-first for the local case.* A Claude Code skill that installs a deterministic PreToolUse hook unifying all four: a running cost/token/call ledger that **hard-denies** (not "ask") at a ceiling; a kill switch via an env-file/flag the hook reads each call (works mid-runaway); a curated prod-aware destructive-op corpus (`rm -rf`, `DROP`/`TRUNCATE`, unscoped `UPDATE`/`DELETE`, force-push, `reset --hard`, kubectl/aws/docker deletes) with dry-run + explicit confirmation; auto prod-target detection (`.env.production`, prod-named DSNs/hosts, kubeconfig contexts) blocked by default. An MCP-wrapper variant extends the same ledger+ruleset to wrapped MCP servers (DB/cloud MCPs) so the boundary also covers tool calls that never touch Bash. Treat a literal "do not run anything" as a hard, session-wide, non-rescindable deny.

**Skeptic's note.** Differentiator vs the installed `/careful` is real (budget metering + hard-deny + kill + prod-block + MCP coverage vs an overridable bash-regex warner). But the dollar anecdotes are single incidents/vendor stats; the *destructive-op* half is the better-evidenced and more clearly valuable leg.

### 4. Retrieval-quality stack — leverage 86
**Problem.** Consolidates RAG retrieval-side failures that masquerade as "the LLM was wrong": chunking that splits a fact or shreds tables/code (~9% recall swing from chunking alone), dense-only recall that misses exact IDs/SKUs/error-codes/acronyms where BM25 would hit every time, and arbitrary single-pass top-K with no reranker that leaves rank-8 evidence out and can't fetch the second hop of a multi-hop question. These are retrieval-engineering problems, not model weaknesses.

**Why high-leverage.** All three constituents high frequency, high/medium severity, and squarely in the tooling layer where a wrapper genuinely fixes them. One retrieval MCP eliminates a whole failure cluster, reusable across every RAG/agent product.

**What already exists & gaps.** Anthropic Contextual Retrieval — a cookbook (not runnable), and contextual embeddings is chunk-enrichment, not structure-aware chunking; no multi-hop. LlamaIndex/LangChain/Haystack/RAGFlow — complete toolkits but builder's kits with naive defaults; easy to misconfigure into exactly these failures. Chonkie/Docling/Unstructured — chunking only. Reranker services + AnswerDotAI `rerankers` — one stage of four. Managed RAG SaaS (Vectara/Ragie/Contextual AI) — all-pillars but closed/hosted/paid, shallow multi-hop. The closest local MCP, **knowledge-rag (lyonzin)**, explicitly hits ~2.5/4: good hybrid + rerank, but falls back to fixed 1000-char windows (no real structure-aware chunking) and has *no* multi-hop loop. RAG "skills" on MCPMarket are advisory prompts, not runtime engines.

**Concrete concept.** *MCP-led "both."* A local-first MCP doing all four with opinionated, pre-tuned defaults: (1) structure-aware chunking (Docling/Chonkie-style table+code+section preservation, not header-only), (2) hybrid BM25+dense+RRF, (3) built-in cross-encoder reranker via the unified `rerankers` API, (4) a bounded iterative multi-hop loop (retrieve → reason → reformulate → 2nd hop → stop). Paired with a thin skill that drives the loop and routes exact-ID/jargon lookups through lexical search. Framing: "knowledge-rag plus the two missing pillars," or "the Contextual Retrieval cookbook turned into a runnable, multi-hop MCP."

**Skeptic's note.** Not a safety issue, so severity caps lower than the top three. The whitespace is specifically *structure-aware chunking + multi-hop, local-first, zero hand-wiring* — frameworks and SaaS already do all four if you're willing to wire/pay.

### 5. Tool-catalogue routing gateway — leverage 84
**Problem.** Three tooling-layer pathologies that worsen with tool count: (1) tool-selection accuracy collapses with catalogue size and near-duplicate tools (fixed 5-tool shortlists found *nothing* on hard queries; adaptive depth raised Claude selection 87.1%→93.1%); (2) silent MCP schema drift — a renamed/retyped param is dropped, the tool "succeeds" on garbage, and the model rationalizes the empty result; (3) doom loops where full-state tool returns re-enter context and re-trigger the same call.

**Why high-leverage.** High freq/severity, sits exactly where a skill/MCP intervention is decisive (interface-and-catalogue, not weights). Pre-filtering also cuts token cost/latency. Capped slightly below the top tier because the residual (a model still mis-picking among top-k, sycophantic victory declarations) is partly model-level.

**What already exists & gaps.** RAG-MCP / Portkey mcp-tool-filter — semantic pre-filter only, fixed-k, no drift/loop handling. mcpindex — pins+diffs in-path and screens tools (the nearest prior art) but no loop damping and does trust-screening, not per-query adaptive retrieval (~1.5/3). mcpdiff/Specmatic — CI-time snapshot/diff, miss the *runtime* silent-success-on-garbage. Invariant/DebounceHook/Hermes doom-loop — loop damping only. MCP gateways (Bifrost, Docker MCP Gateway, AgentCore) — treat drift as something to *absorb* via versioned transformation (the opposite of alarming on it); three concerns live in three subsystems. Adaptive-depth (arXiv 2605.24660) — a measurement paper, unshipped.

**Concrete concept.** *MCP-led "both."* One in-path proxy that on every `tools/list` does query-**adaptive-depth** semantic retrieval (recovering the ~6pp lost to fixed shortlists), and on every `tools/call` (a) diffs live response vs pinned schema and emits a **loud, agent-visible** alarm on contract-valid-but-empty/garbage results, and (b) fingerprint-damps repeated identical calls. The defensible core is the cross-signal link nobody has built: feed the runtime-drift signal and the loop signal into each other so a doom loop caused by silent drift is attributed to drift and broken with a corrective message, not blindly throttled. Thin skill adds query-difficulty-aware routing and forced-backtrack-after-N.

**Skeptic's note.** The adaptive-depth lift (87.1→93.1) rests on a single paper. The genuinely novel piece is the drift↔loop cross-attribution; the rest is recombining existing legs.

### 6. Cross-session memory with temporal versioning + contradiction reconciliation — leverage 82
**Problem.** Statelessness (agents re-meet the user each session, repeat resolved mistakes) plus the harder second-order problem: naive memory becomes *confidently wrong*. Stale high-confidence facts stay retrievable after they stop being true ("confidently wrong about your employer after you change jobs"); append-only stores accumulate contradictions; dedup/temporal-versioning/source-precedence/retirement is an explicitly open problem.

**Why high-leverage.** High/high and almost entirely tooling-shaped — persistence and consolidation are infrastructure, not model behavior. Durable memory is a horizontal capability every agent needs and most ship without; bundling consolidation from day one avoids the well-documented "bolt-on memory creates a worse staleness problem" trap.

**What already exists & gaps.** **This one has a genuine exception to the "all partial" pattern: Zep/Graphiti already implements bi-temporal versioning + contradiction-as-invalidation + provenance, and ships an MCP server — so don't reinvent temporal-graph memory.** But Graphiti is heavy (Neo4j/FalkorDB, ~600k tokens/conversation in one comparison vs ~1.7k for Mem0) and *eventually-consistent* (immediate post-ingestion retrieval often fails), breaking the write-at-end/read-at-start latency assumption — and even it lacks an explicit source-precedence ruleset and confidence-decay-on-staleness. Every other shipping layer (Mem0, LangMem, Letta, Anthropic memory tool + reference KG server, Basic Memory/claude-mem) is destructive last-write-wins or append-only — they solve recall, not coherence. 2026 benchmarks (LOCOMO + contradiction-resolution categories) *measure* coherence but don't solve it.

**Concrete concept.** *MCP-primary.* A lightweight, synchronous, dependency-light "coherence layer" over the now-commodity recall substrate (or self-contained over SQLite) adding the four under-served operations: (1) temporal versioning with explicit valid-from/valid-to so old values are superseded not deleted; (2) source attribution + a configurable source-precedence policy (e.g. user-stated > agent-inferred > document-extracted) to decide which conflicting fact wins; (3) contradiction *detection* that flags at write time (rather than silently replacing), with an optional review queue; (4) staleness-aware retirement on TTL/supersession. Differentiator vs Mem0: non-destructive versioning + provenance precedence. Vs Graphiti: synchronous, light, install-in-one-line, policy-explicit. A thin skill encodes the write-salient-facts / verify-before-asserting discipline.

**Skeptic's note.** This is the one issue where a strong incumbent (Graphiti) genuinely does most of the hard thing; the opportunity is narrower (lightweight + synchronous + explicit-policy), so size it honestly against "just run Graphiti."

### 7. Indirect prompt-injection / untrusted-tool-output containment — leverage 80
**Problem.** Indirect prompt injection via tool output (attacker text in web pages, emails, GitHub issues, PDFs, or MCP tool descriptions hijacks the agent's goal; reported 50-85% ASR across frontier models; Invariant's single-issue private-repo exfil PoC) plus the confused-deputy/over-trusting-output problem (tool poisoning, rug pulls, shadowing — the LLM holds the union of all privileges and acts on attacker-controlled data as authoritative). Mitigation: Dual-LLM/quarantine — untrusted content never reaches the privileged action-taking context; tool output/descriptions tagged untrusted, pinned, instruction-stripped.

**Why high-leverage.** Severity maximal (exfiltration, goal hijack), frequency high. Architectural mitigation is skill/MCP-implementable even though the underlying instruction-vs-data confusion is model-level. Scored a notch below the cost/destruction governor because no wrapper fully closes it — it reduces, not eliminates, ASR.

**What already exists & gaps.** The trifecta splits into two halves. **The MCP tool-description half is well-covered by shipped tools — MCP-Scan + tool pinning and ETDI signing directly address poisoning/rug-pull/shadowing; building there would mostly reinvent.** The untrusted-tool-*output* half is the gap: every production option is either (a) probabilistic detection that still lets untrusted content reach the privileged context (LlamaFirewall/PromptGuard-2 — recall as low as 0.40 on real attacks, Lasso, Lakera, spotlighting/datamarking), or (b) true architectural containment that exists only as explicitly-not-for-production research (CaMeL — google-research warns the interpreter "likely contains bugs"; Microsoft Dromedary README says "NOT TO BE USED IN PRODUCTION") requiring a full agent rewrite.

**Concrete concept.** *MCP-led "both."* A "quarantine proxy" MCP/middleware in front of content-fetching tools (web, email, GitHub, PDF, other MCP servers) that (1) routes raw output to a sandboxed quarantined sub-LLM that extracts only structured, schema-validated fields — never free-form instructions — returning opaque symbolic handles to the privileged agent; (2) deterministically normalizes/instruction-strips first (Unicode/zero-width/homoglyph cleaning, HTML stripping, datamarking) before any model sees it; (3) enforces a per-handle egress capability check at action time so data tagged untrusted/private can't flow to an external-communication tool without an explicit policy/human gate — collapsing the lethal trifecta to "pick two." Pairs naturally with MCP-Scan/ETDI (which guard the description half it doesn't).

**Skeptic's note.** Honest framing: this *reduces* ASR, never zeroes it, and the quarantine sub-LLM is itself an attack surface. The 50-85% ASR figure spans a survey range; the value proposition is the architectural containment that no shipped artifact currently packages, not a clean "fix."

### 8. Execution-robustness wrapper — leverage 75
**Problem.** Bundles the resilience gaps that turn ordinary API friction into agent failures: retrying non-retryable errors (401/400) in tight token-burning loops while mishandling transient ones (429/503/timeouts) that needed backoff; processing only page 1 and declaring complete; breaking on non-JSON/legacy error bodies; and the history-poisoning amplifier where one malformed tool-call message gets re-serialized and bricks the whole session with permanent HTTP 400.

**Why high-leverage.** Frequency high; practitioners report these "unhandled external failure" problems cause more production breakage than agent-logic bugs. Scored at the lower end of the top tier because severity is mostly medium and one constituent (the model emitting valid JSON in the first place) is intrinsically model-level — the wrapper prevents downstream poisoning, not the original malformed emission.

**What already exists & gaps.** Backoff is a solved primitive (tenacity/stamina/backoff). LangChain tool_retry/LangGraph ToolNode — exception-based, not HTTP-status-classified, no idempotency, no pagination, no validate-before-append; robust path off by default in LangGraph >1.0.1. AI gateways (LiteLLM/Portkey/Bifrost) — operate on the *LLM call*, not per-tool I/O. MCP pagination — server-side only; client-side `nextCursor` following is left to the LLM (the exact "page 1 = complete" failure). Idempotency/durable-execution (Temporal) — heavy architectural commitment. ALTK/MS Agent Framework/SPARC — closest to history-repair but framework-bound. **The standout open gap: validate-before-append / tool-message history repair is being independently rediscovered and one-off-patched in kimi-cli, hermes-agent, openclaw, jan — no shared reusable component.**

**Concrete concept.** *MCP-led "both"; MCP-proxy is the higher-leverage form.* A framework-agnostic wrapper that (1) classifies tool failures by HTTP status/error body into retry vs abort, (2) owns exponential backoff + jitter + Retry-After + idempotency keys auto-applied only to mutating calls, (3) auto-paginates by detecting cursor/Link/offset patterns and looping to exhaustion before returning one consolidated result, (4) normalizes non-JSON/legacy bodies into a canonical shape, (5) validates-and-repairs every tool-call/tool-result message (`json.loads` the args, coerce truncated/invalid args, fix orphaned `tool_call_ids`) before append. As an MCP proxy it neutralizes the history-poisoning amplifier for any agent without per-framework patching.

**Skeptic's note.** The validate-before-append leg is the best-evidenced (a named, repeatedly-rediscovered bug across multiple real repos) and the most defensible reason to build; the rest is competent recombination of known primitives.

## 4. Model-Level vs Solvable (manage expectations)

**Genuinely addressable by a skill/MCP (the eight above)** because they live in the harness/tooling layer or convert a model weakness into an external check: identifier existence, completion verification, runaway/destructive governance, retrieval quality, tool routing/drift/loops, cross-session memory coherence, untrusted-output containment, execution robustness. Also meaningfully *mitigable* (work-around, not cure): ungrounded-claims/citation-faithfulness (skill enforcing claim-to-span checks + abstention), lossy compaction/goal-drift/context-rot (skill re-anchoring + offload), deprecated-API conflict (MCP serving live docs), under-calling tools (skill forcing action-selection), KV-cache fragility (cache-first prompt structuring), cost attribution and routing collapse (MCP proxy tagging/ranking).

**NOT skill/MCP-fixable — a wrapper can only work around, never cure (set expectations here):**
- **Systematic overconfidence / miscalibration** — intrinsic to weights/RLHF. A skill can force consider-alternatives or multi-sample agreement as a partial proxy; true calibration is not addressable. *(vehicle: neither)*
- **Effective context far smaller than advertised; reasoning degrades long before the token limit** (NoLiMa: 11/12 models <50% of short-context perf at 32K) — architecture/attention. Skills can only keep payloads small/chunked. *(neither)*
- **Lost-in-the-middle position bias** — architectural; no production model has eliminated it. Reorder-to-edges helps at the margin. *(skill = work-around only)*
- **Malformed tool-call argument *emission* (bad JSON/types)** — a generation defect; best fixed by constrained/structured decoding + `strict:true` at the provider. A harness can only prevent the downstream history-poisoning amplifier (see #8), not the bad emission. *(neither for emission)*
- **Multi-turn jailbreaks (Crescendo, many-shot)** — model alignment property; needs safety training + a provider-side conversation-level classifier. *(neither)*
- **Agentic misalignment / reward-hacking / eval-gaming / goal-preservation** — training/alignment problem; external monitors help only at the margins. *(neither)*
- **Per-step errors compounding over long horizons + self-conditioning + non-linear horizon collapse** — the underlying per-step accuracy and self-conditioning tendency are intrinsic and *not fixed by scale*. A skill caps the *effective* horizon via decomposition/checkpointing but cannot raise the per-step ceiling. *(skill = partial mitigation, honestly model-bound)*
- **Reasoning-mode TTFT latency** — a model property; skills cut round-trips but not core thinking-token latency. *(skill = partial)*
- **Non-determinism of eval signals** — inherent; a skill can impose a statistically-grounded protocol (pin seed/temp/version, N samples, CIs) but cannot make runs deterministic. *(skill = protocol only)*

**The honest headline:** the most dangerous failure (overconfidence — "no signal when the model is guessing") is in the *unfixable* column. That is precisely why the #1 opportunity matters — it sidesteps calibration by replacing "is the model confident?" with "does the referent exist?", a question that doesn't require the model to know what it doesn't know.

## 5. Recommended Next Step (research directions, ranked — not a build go-ahead)

1. **Validate the cross-class identifier-verifier (rank 1) as a single in-loop gate.** *Why start here:* highest leverage in the corpus, the only deterministic ("existence is a binary lookup") candidate, and the package/path/tool-name slice has genuinely *no* agent-callable verifier today — a clean whitespace where a small probe (does an MCP `verify_identifiers` resolver run fast enough to gate every turn, and does returning canonical/redirected referents handle the messy cases?) would de-risk the whole thesis cheaply.

2. **Prototype the acceptance-criteria + ground-truth gate (rank 2) by wrapping the existing Stop hook.** *Why start here:* the principle is the most strongly evidenced (incorrect verification = strongest single predictor of long-horizon failure), the substrate already exists locally (Stop hook + Rigour to differentiate against), and the only real unknown is adoption — whether users will register criteria up front — which a lightweight experiment answers before any heavy investment.

3. **Scope the destructive-action governor (rank 3) as an upgrade path from the already-installed `/careful` skill.** *Why start here:* maximal severity, the clearest "guarantee not suggestion" case, and the gap is concrete and bounded (add budget metering + hard-deny + kill switch + prod-block to a tool that today only warns) — meaning the differentiator can be specified precisely before committing, and the destructive-op half is better-evidenced than the dollar-spiral anecdotes it's often bundled with.

*Deliberately not in the top three: memory coherence (rank 6) because Graphiti already does the hard part, and the containment proxy (rank 7) because it only reduces (never closes) ASR and its description-half is already shipped — both are real but worse first bets than the three above.*
